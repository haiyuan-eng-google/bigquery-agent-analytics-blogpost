## Problem Statement

BigQuery is the most common data platform for AI agent analytics, but its CLI tooling (`bq`) was designed in 2012 for human operators. It is not extensible (monolithic Python binary, no plugin system), not agent-friendly (inconsistent output formats, no structured JSON default, verbose help that wastes context tokens), and not AI-aware (no integration with Conversational Analytics, AI functions, or agent evaluation).

Meanwhile, AI coding agents (Claude Code, Gemini CLI, Cursor, Codex CLI) are becoming primary consumers of CLI tools. Benchmarks show CLIs are 10-32x more token-efficient than MCP, with 100% reliability vs 72% for MCP (Scalekit, 2026). But agents need CLIs designed for them: structured output, progressive disclosure, and discoverable skills.

dcx is an agent-native CLI for Google Cloud's Data Cloud that combines dynamic command generation from Discovery APIs, BigQuery Agent Analytics SDK capabilities, and Conversational Analytics API integration into a single Rust binary.

## Demand Evidence

- BQ Agent Analytics Plugin: 160M events/month, 600K BQ jobs, 300B events scanned (real external users)
- CLI vs MCP benchmarks: 10-32x cheaper, 100% reliable, 33% better token efficiency
- `gws` CLI precedent: proved agent-native CLI model works for Google Workspace, adopted by Claude Code, Gemini CLI, Cursor
- `bq` CLI has not been meaningfully updated for the agent era
- No competitor is building an agent-native CLI for Data Cloud (everyone competes on warehouse engines, not the interface layer)
- Internal team using dcx for dogfooding (but zero external users yet)

## Status Quo

**Humans:** Use `bq` CLI + BigQuery console + custom Python scripts. Write raw SQL for analytics. Use Looker Studio dashboards for visualization. No natural language query capability from the terminal.

**AI agents:** Invoke `bq` commands or BigQuery MCP servers. MCP tools inject 44K tokens of schema per conversation (vs 1.4K for CLI). Agents can't discover BQ capabilities through SKILL.md. No structured error handling. No progressive disclosure.

**Combined cost:** Every team builds their own analysis scripts, eval pipelines, and agent integrations from scratch. No shared, discoverable tooling layer.

## Target User & Narrowest Wedge

**Primary target:** Developers using AI coding agents (Claude Code, Gemini CLI, Cursor) who need their agents to interact with BigQuery, evaluate AI agent sessions, or query data in natural language.

**Secondary target:** Platform/ML engineers who operate BigQuery from terminals and want agent analytics capabilities without writing SQL.

**Wedge:** The Agent-First Bundle: ship dcx with analytics + CA + SKILL.md, targeting AI agent users specifically. The value prop: "Install dcx, your AI agent can now query BigQuery, evaluate agent sessions, and ask natural language questions."

## Constraints

- Rust single binary, distributed via npm (`npx dcx`, `npm install -g dcx`)
- Cross-platform: macOS ARM64/x64, Linux x64/ARM64, Windows x64/ARM64
- JSON-first output (+ table, text formats)
- SKILL.md format for agent discoverability (compatible with Claude Code, gws pattern)
- CLI-first architecture (MCP can be added later if protocol matures)
- Must work with Application Default Credentials (gcloud auth)
- Dynamic commands are read-only (no mutation via Discovery API)
- CA API is in preview, may have availability constraints
- **Error handling:** All commands return structured JSON errors: `{"error": true, "code": "AUTH_FAILED", "message": "...", "suggestion": "Run gcloud auth application-default login"}`. Exit code 2 for infrastructure errors (auth, network, API), exit code 1 for evaluation failures (threshold exceeded). Agents can parse error codes programmatically.
- **Auth failure path:** If ADC credentials are missing or expired, dcx prints a structured error with `code: AUTH_FAILED` and `suggestion` field pointing to the fix. No stack traces. No ambiguous messages.
- **CA API fallback:** If the CA API returns errors >5% of requests during beta, `dcx ca ask` logs the failure structured and suggests using `dcx analytics` commands (which hit BigQuery directly, not the CA API) as a fallback for agent analytics use cases. The SKILL.md for dcx-ca includes a `limitations` section noting preview status.
- **Plugin dependency for analytics:** `dcx analytics` commands require the BQ Agent Analytics Plugin to be active in the target BigQuery project (server-side dependency: the `agent_events` table must exist with data). `dcx analytics doctor` checks this prerequisite and returns a clear error if the table is missing.

## Premises

1. **The CLI interface is the new battleground.** In an agent-first world, AI agents consume CLIs 10-32x more efficiently than MCP. dcx's agent-native design is a structural advantage over bq CLI that can't be retrofitted. (Note: the second opinion challenged this as a timing play, not a permanent moat. We keep the premise but acknowledge MCP may close the gap within 12 months.)
2. **dcx's broadest wedge is `dcx ca ask` (natural language queries), but the launch targets AI agent users.** CA works with ANY BigQuery dataset (10M+ users). Analytics requires the plugin (subset). For the Agent-First Bundle, both are needed.
3. **dcx needs external users before it's a product.** Internal dogfooding is necessary but not sufficient. Until an external user types `npx dcx` and gets value, this is a tool, not a product.
4. **Rust + single binary + npm distribution is the right technical choice.** Fast startup, no Python dependency, cross-platform, matches how CLI agents get distributed.
5. **The biggest risk is positioning, not engineering.** dcx does 6 things. Users need to understand why they'd install it in one sentence.

## Approaches Considered

### Approach A: The CA Wedge
Ship `dcx ca ask` only. Validate with 5 external users. Completeness: 4/10.

### Approach B: The Agent-First Bundle (CHOSEN)
Ship analytics + CA + SKILL.md targeting AI agent users. 2-3 weeks. Completeness: 7/10.

### Approach C: Full Launch
All 14 skills with positioning fix. 4-6 weeks. Completeness: 8/10.

## Recommended Approach

**Approach B: The Agent-First Bundle.** Target AI coding agent users first. Ship three capabilities that bq CLI can never offer.

### What ships

| Capability | Commands | Why it matters for agents |
|---|---|---|
| Conversational Analytics | `dcx ca ask`, `dcx ca create-agent`, `dcx ca list-agents` | Natural language to SQL. Agents don't need to generate BQ SQL. |
| Agent Analytics | `dcx analytics evaluate` (gate sessions against latency/error/cost thresholds), `dcx analytics get-trace` (reconstruct a session's event DAG), `dcx analytics doctor` (verify table schema and data freshness), `dcx analytics drift` (compare production query distribution against a golden question set to find coverage gaps), `dcx analytics insights` (generate a multi-section analysis report: task areas, friction points, tool usage, trends) | Agents can self-evaluate, debug traces, detect drift. No other CLI does this. |
| SKILL.md discoverability | 3 SKILL.md files (dcx-ca, dcx-analytics, dcx-bigquery) | AI agents auto-discover capabilities. Progressive disclosure. Token-efficient. |

### What doesn't ship yet

- Dynamic commands for Spanner, AlloyDB, Cloud SQL, Looker (defer to post-validation)
- Recipe skills (recipe-debugging, recipe-quality-ops, recipe-source-onboarding)
- MCP server mode

### Execution strategy: validate CA first, ship the bundle

The bundle includes three capabilities, but the go-to-market hinges on `dcx ca ask` being the entry point. Validation sequence: first confirm CA works for external users (the wedge), then ship the full bundle with analytics and SKILL.md as the retention loop.

### Phase 1: CA Validation + Agent Polish (Week 1)

1. **Validate CA end-to-end:** Run `dcx ca ask` against 3 different real BigQuery datasets. Measure: does it return a useful answer in <30 seconds? If CA error rate >5%, fix or file upstream before proceeding.
2. **Define SKILL.md test protocol:** A SKILL.md passes validation when an AI agent (Claude Code) can read it, discover the available commands, execute a 3-step workflow (doctor → evaluate → get-trace), and get structured JSON results without human intervention. Test by adding dcx skills to a fresh CLAUDE.md and running a scripted conversation.
3. Write a Claude Code integration guide: how to add dcx skills to a project's CLAUDE.md
4. Write a Gemini CLI integration guide
5. Verify `npx dcx` works on first try on macOS ARM64 and Linux x64 (the two most common dev platforms; defer other platforms to post-beta)
6. Record a demo GIF: Claude Code using dcx to evaluate an agent session

### Phase 2: External Beta (Week 2)

1. Identify 7 target users (draft outreach messages before Phase 1 ends):
   - 3 from GitHub discussions (BQ Agent Analytics Plugin users who wrote custom scripts)
   - 2 Claude Code power users (search GitHub for CLAUDE.md files referencing BigQuery)
   - 2 Gemini CLI users
2. Ship via `npx dcx` (end-to-end tested in Phase 1)
3. Measure via a 5-question follow-up form: (a) time from `npx dcx` to first useful output, (b) which command they tried first, (c) which command was most useful, (d) what they tried that didn't work, (e) would they use it again next week
4. Track: command usage via opt-in telemetry (structured, local-first, same pattern as gstack)

### Phase 3: Iterate (Week 3+)

Based on beta feedback:
- If CA is the pull: double down on `dcx ca ask`, add more data sources
- If analytics is the pull: align with the SDK query-log analysis from the prior design doc
- If SKILL.md discoverability is the pull: add more skills, expand to Spanner/AlloyDB
- If nothing pulls: the product thesis needs revision

## Open Questions

1. **CA API availability:** Is the Conversational Analytics API stable enough for external users? Preview APIs can break.
2. **Auth flow for npx (MUST RESOLVE IN PHASE 1):** Does `npx dcx ca ask` pick up ADC credentials seamlessly on first run? Test: fresh macOS terminal with `gcloud auth application-default login` done, run `npx dcx ca ask "show tables" --project=test-project`. If it fails, add `dcx auth login` as a first-run wrapper. This is a first-run-experience blocker and cannot be left open through beta.
3. **Positioning one-liner:** What's the one sentence? Candidates:
   - "The AI-native BigQuery CLI. Ask in English. Evaluate your agents. Replace bq."
   - "dcx: what bq would be if it were built for AI agents."
   - "Talk to BigQuery. Debug your agents. One CLI."
4. **gws relationship:** Is dcx a sibling of gws (same pattern, different domain) or does it eventually merge into gws?
5. **Binary size:** How big is the dcx binary? If >50MB, `npx` cold-start may be slow.

## Success Criteria

- Phase 1: 3 SKILL.md files that AI agents can use without human intervention
- Phase 2: 5+ external users try dcx. At least 3 get a useful result in <60 seconds.
- Phase 3: Identify which of the three capabilities (CA, analytics, discoverability) has the strongest pull
- 6-month: dcx referenced in 10+ Claude Code CLAUDE.md files in public repos
- 12-month: dcx adopted by at least one AI coding agent framework as a recommended BQ integration
