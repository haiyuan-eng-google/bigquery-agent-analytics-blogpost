## Uploading to Medium

1. In the Medium editor, click the `+` at each placeholder position and select the image.
2. Medium auto-scales; these are rendered at roughly 2× desktop width so they stay crisp on retina.
3. Add alt-text matching the "What it shows" column above (accessibility + SEO).
4. Recommended placement:
   - `05_hero_rows_to_tree.png` as the **cover image** on the Medium post settings panel.
   - `01_hook_raw_agent_events.png` inserted right after the opening hook paragraph in section 1.
   - `02_doctor_output.png` right after the `bq-agent-sdk doctor` code block.
   - `03_trace_render.png` right after the `trace.render()` text output block in section 4 (optional — skip if the textual render feels enough).
   - `04_information_schema.png` right after the SQL code block in section 6.

## Regenerating

Any of these can be regenerated by editing `generate.py` and re-running:

```bash
python3 generate.py
```

All five rebuild from the constants in the script. If the demo is
re-run against a different dataset or model, update the constants
(session ID, latencies, aggregation numbers) at the top of each
`shot_*` function to match the new captures.

## What these are NOT

These are high-fidelity renders in the **style** of BQ console and
terminal screenshots — not literal pixel captures of those tools.
Every number, session ID, event type, and latency is real, pulled
from the live run; the chrome (window dots, URL bar, query editor
panel) is stylized for consistency across the post. If the
publication's standards require native screenshots, regenerate by
actually taking them from the BQ console and a terminal session
against the same data; the filenames and placements above stay the
same, the images just get swapped.
