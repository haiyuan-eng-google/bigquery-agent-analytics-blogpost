## Problem Statement

The BigQuery Agent Analytics Plugin (shipping in Google's ADK since v1.21.0) captures AI agent lifecycle events into BigQuery. It has real production traction: 160M events/month, 600K BQ jobs, 300B events scanned in trailing 30 days. The write side is solved.

The read side is fragmented. Users analyze their agent data through raw SQL queries, custom Python scripts, and a Looker Studio dashboard template. There is no programmatic SDK for common analysis patterns. Every team rebuilds the same trace reconstruction, latency analysis, and eval logic from scratch.

The opportunity is a Python SDK (`bigquery-agent-analytics`) that turns the most common ad-hoc query patterns into one-line function calls.

## Demand Evidence

- 160M events written per month via the plugin (real production usage, not test data)
- 600K BigQuery jobs in 30 days = users actively querying and analyzing agent event data
- 300B events scanned = heavy read workloads against the plugin's table schema
- GitHub discussion google/adk-python#3951: user (@Dustan-FullStory) spent days debugging Agent Engine deployment, demonstrating reliance on the plugin for production agent observability
- Users currently build analysis using: raw SQL in BQ console, custom Python scripts, Looker Studio dashboards
- Plugin ships inside Google's ADK — distribution is solved

## Status Quo

Users write raw SQL to answer questions about their agents. Common patterns observed through GitHub issues and the auto-created views (v_llm_request, v_tool_completed, etc.):

- Trace reconstruction: `WHERE trace_id = X ORDER BY timestamp`
- Latency analysis: `AVG(CAST(JSON_VALUE(latency_ms, '$.total_ms') AS INT64))`
- Error analysis: `WHERE event_type IN ('LLM_ERROR', 'TOOL_ERROR')`
- Token usage: `JSON_VALUE(content, '$.usage.total')`
- Tool provenance: `GROUP BY JSON_VALUE(content, '$.tool_origin')`

These patterns are repeated across every team that adopts the plugin. The SDK's job is to eliminate this redundancy.

## Target User & Narrowest Wedge

**Target user:** A platform/ML engineer at a company running AI agents on GCP (using ADK with the BQ Agent Analytics plugin). They've already set up the plugin and have events flowing into BigQuery. They write SQL and Python to analyze agent performance but don't want to maintain custom analysis scripts.

**Narrowest wedge:** Before building SDK features based on intuition, analyze the 600K BQ jobs from the last 30 days using `INFORMATION_SCHEMA.JOBS`. Extract the top 20 SQL query patterns. Cluster them by intent (trace lookup, latency analysis, error investigation, token budgeting, etc.). Build the top 5 patterns as SDK functions.

## Constraints

- Must work with the existing plugin table schema (agent_events, with partitioning by timestamp, clustering by event_type/agent/user_id)
- Must be compatible with the auto-created per-event-type views (v_llm_request, v_tool_completed, etc.)
- Python package, publishable to PyPI
- Minimal dependencies: `google-cloud-bigquery` + optionally `rich` for CLI rendering
- Open-source, Apache 2.0, companion to the ADK plugin
- BigQuery-native — no separate infrastructure required for the 80% use case (batch analytics, eval pipelines)
- For real-time trace lookup (the 20% use case), may need a serving layer consideration at scale (materialized views, BI Engine, or Bigtable cache)
- **Authentication:** SDK client accepts an explicit `google.cloud.bigquery.Client`, or initializes one from Application Default Credentials (ADC). No custom auth layer.
- **Schema versioning:** SDK must handle tables created by different plugin versions (pre-1.26.0 tables may lack `tool_origin` column, pre-1.27.0 may lack auto-created views). Functions should degrade gracefully with missing columns, not crash.
- **Error handling:** Wrap BQ API errors (quota, permission, not-found) into SDK-specific exceptions with actionable messages (e.g., "Table not found — ensure the BQ Agent Analytics Plugin is configured and has written at least one event").

## Premises

1. The write side (plugin) is solved with real distribution (ADK) and real usage (160M events/month).
2. The read side (SDK) is the product opportunity — users already do this work with raw SQL and custom scripts.
3. Competitive advantage is BigQuery-native for analytics and batch evaluation. For real-time debugging (single trace lookup), BQ may need a serving assist at scale. "Zero new infrastructure" holds for batch workflows but may not hold for live debugging at extreme scale.
4. The SDK should be an open-source companion to the plugin, distributed through Google's ADK ecosystem.
5. The biggest risk is scope: 12 features in the current SDK design is a platform, not a wedge. Users will adopt feature-by-feature.

## Approaches Considered

### Approach A: The Wedge
3 features only (trace, stats, diff). CLI-first. Ship in 1-2 weeks to 3-5 power users. Completeness: 3/10.

### Approach B: The Full Platform
All 12 features as designed in the SDK doc. 3-4 months. Completeness: 10/10. Risk: building features nobody uses.

### Approach C: Query-Log Driven (CHOSEN)
Analyze 600K real BQ jobs first. Extract top 20 SQL patterns. Build the top 5 as SDK functions. 3-4 weeks. Completeness: 6/10. Every feature backed by observed demand.

## Recommended Approach

**Approach C: Query-Log Driven.** Let the data tell you the product.

### Phase 1: Query Pattern Analysis (Week 1)

Run analysis against `INFORMATION_SCHEMA.JOBS` to extract and cluster query patterns hitting the `agent_events` table:

```sql
SELECT
  query,
  total_bytes_processed,
  creation_time,
  user_email
FROM `region-us-central1`.INFORMATION_SCHEMA.JOBS
WHERE
  creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
  AND state = 'DONE'
  AND statement_type = 'SELECT'
  AND EXISTS (
    SELECT 1 FROM UNNEST(referenced_tables) AS t
    WHERE t.table_id = 'agent_events'
  )
ORDER BY creation_time DESC
LIMIT 10000;
```

**Important limitation:** `INFORMATION_SCHEMA.JOBS` is project-scoped. You can only see queries in projects you have `bigquery.jobs.list` permission on. Since plugin users run agents in their own projects, this analysis is limited to projects you control. Supplement with: (a) GitHub issue/discussion analysis to infer patterns, (b) voluntary telemetry opt-in for query pattern categories, (c) your own team's heaviest-usage project as a representative sample.

Cluster queries by intent using regex pattern matching on SQL clauses (WHERE filters, GROUP BY, JSON_VALUE paths), validated by manual review of a 200-query sample:
- Trace reconstruction (WHERE trace_id/session_id = X)
- Latency/performance analysis
- Error investigation
- Token usage / cost analysis
- Tool provenance analysis
- Time-series trends
- Other

Quantify: what % of 600K jobs fall into each cluster?

### Phase 2: SDK Core (Weeks 2-3)

Build SDK functions for the top 5 query patterns. Each function:
- Wraps a parameterized SQL template
- Returns structured Python objects (not raw BQ rows)
- Supports both programmatic use and CLI output
- Has type hints, docstrings, and basic tests

Hypothesized candidates (to be confirmed by Phase 1 data). Decision rule: if a hypothesized function does not appear in the top 10 query patterns, it is cut from Phase 2.

1. `client.get_trace(session_id)` — trace reconstruction with nested timeline (renders event DAG as a tree)
2. `client.stats(agent_id, window)` — latency/cost/token aggregates over a time window
3. `client.errors(agent_id, window)` — error analysis with surrounding event context
4. `client.diff(session_a, session_b)` — diff of event sequences and tool call chains between two traces (compares event types, tool names, and args side-by-side, highlighting divergence points)
5. TBD by data — whatever the 5th most common pattern is

### Phase 3: Validation (Week 4)

- Publish to PyPI as `bigquery-agent-analytics`
- Share with 3-5 most active users from GitHub issues/discussions
- Measure: which functions get used most? What's the first feature request?
- That request becomes the roadmap for Phase 4

### Phase 4: Iterate (Ongoing)

Build features based on observed usage. The full SDK doc (evals, LLM-as-judge, drift detection, memory) becomes the long-term vision, but each feature ships only when demand is confirmed.

## Open Questions

1. **Access to INFORMATION_SCHEMA.JOBS:** Can you query job metadata across projects to see what SQL patterns users are running against agent_events tables? Or is this limited to your own project?
2. **Latency benchmarks:** What is the P95 latency for single-trace reconstruction at 160M events/month? This determines whether a serving layer is needed.
3. **Package naming:** `bigquery-agent-analytics` on PyPI — is this name available and approved?
4. **Relationship to ADK:** Does the SDK ship inside the ADK package or as a separate companion package?
5. **Cross-project support:** Users like @Dustan-FullStory log to a different project than where agents run. Does the SDK need cross-project query support?

## Success Criteria

- Phase 1: Identify the top 5 query patterns with confidence (>80% of jobs covered)
- Phase 2: SDK published to PyPI, installable via `pip install bigquery-agent-analytics`
- Phase 3: 3+ users adopt the SDK and replace their custom SQL scripts
- Phase 4: Feature requests come from SDK users, not from the builder's intuition
- Long-term: 50%+ of the 600K monthly BQ jobs shift from raw SQL to SDK calls
